# Companion Scheduling — デバッグ・受入テスト要件定義書

**目的:** 複数名で並行してデバッグする際に、「何を」「どの条件で」「どうなれば合格か」を揃えるための文書です。  
**対象:** 本番相当環境（Firebase バックエンド推奨）  
**更新目安:** 機能追加・プラン変更のたびに該当セクションを更新する

---

## 1. この文書の使い方


| 記号       | 意味                       |
| -------- | ------------------------ |
| **必須**   | リリース前に全員または担当ロールで確認する    |
| **推奨**   | 時間があれば確認。関連機能を触ったときは必ず実施 |
| **前提**   | テスト開始前に満たす条件             |
| **期待結果** | これが満たされなければ不具合（バグ）として記録  |


**進め方の例**

1. セクション 2 で環境とテストデータを準備する
2. セクション 3 の「共通スモーク」を全員が一度通す
3. 担当を分けてセクション 4〜6 を実施（重複してもよい）
4. 不具合はセクション 8 の形式で共有する

---

## 2. テスト環境・前提

### 2.1 環境


| 項目     | 要件                                                            |
| ------ | ------------------------------------------------------------- |
| バックエンド | **本番と同じ `DATA_BACKEND=firebase`** を推奨（Prisma/SQLite はローカル開発用） |
| メール    | 実際に届くアドレスで確認（通知・予約確定・ロールプレイ開示）                                |
| タイムゾーン | 既定 `Asia/Tokyo`。企業設定で上書きした場合はその TZ で確認                        |
| ブラウザ   | Chrome / Safari のいずれか。クライアント・パートナーは別ブラウザ or シークレット推奨          |


### 2.2 ロール一覧


| ロール                 | 主な画面                         | 書き込み権限の目安                        |
| ------------------- | ---------------------------- | -------------------------------- |
| **ADMIN**           | `/admin/`* すべて               | すべて可能                            |
| **ADMIN_ASSISTANT** | ADMIN と同じ画面                  | **閲覧＋チャットコメントのみ**。設定変更・請求確定などは不可 |
| **PARTNER**         | ホーム、マッチルーム、請求書、会議リンク設定       | 日程提案・確定、パートナーレポート等               |
| **CLIENT**          | ホーム、マッチルーム                   | 日程投票、振り返り、コーチング入力等               |
| **CLIENT_ADMIN**    | CLIENT ＋ 自社セッション一覧・スキルチェック俯瞰 | 他社員のルームには入れない（自分がクライアントのマッチのみ）   |
| **CLIENT_HR**       | CLIENT_ADMIN と同等             | 同上                               |


### 2.3 導入プラン（3種）


| プラン ID                         | 表示名           | ざっくりした違い                                   |
| ------------------------------ | ------------- | ------------------------------------------ |
| `workplace_activation`         | 職場活性プラン       | チャット・日程・1on1・自分FTA                         |
| `individual_companion`         | 個別伴走プラン       | 上記＋成果物（スキルチェック、PDCA、振り返り等）。企業ごとに ON/OFF 可  |
| `coaching_management_training` | コーチングマネジメント研修 | アイスブレイク・質問リスト・1on1フォーマット・ロールプレイ評価。自分FTA なし |


プランは **アプリ設定 → 企業（テナント）登録** で企業ごとに設定する。  
クライアントの `companyId` が未設定だとプラン機能が正しく効かない。

## 3. テストデータ準備（最小構成）

以下の **8 パターン** を用意すると、ほぼすべての要件をカバーできます。  
担当者が被らないよう、ID（メール）をチーム内で割り当ててください。


| ID     | 企業プラン   | ユーザー                  | マッチ状態          | 主なテスト目的             |
| ------ | ------- | --------------------- | -------------- | ------------------- |
| **T1** | 職場活性    | P1, C1                | パートナー割当済       | 基本フロー（チャット・日程・1on1） |
| **T2** | 個別伴走    | P2, C2                | 割当済・成果物すべて ON  | FTA・スキルチェック・PDCA 等  |
| **T3** | 個別伴走    | P3, C3                | 割当済・FTA だけ OFF | 企業ごとの成果物 ON/OFF     |
| **T4** | コーチング研修 | C4 のみ                 | **パートナー未割当**   | 研修ルーム先行オープン         |
| **T5** | コーチング研修 | P4, C4                | 割当済            | ロールプレイ・相互開示・請求      |
| **T6** | 任意      | CA1（CLIENT_ADMIN）, C5 | C5 のみマッチ       | 管理者向け一覧（パートナー名非表示）  |
| **T7** | 任意      | AA1（ADMIN_ASSISTANT）  | —              | 書き込み拒否の確認           |
| **T8** | 任意      | C6                    | 企業未割当          | 管理者警告・フォールバック       |


### T4 の作り方（コーチング・パートナー未割当）

1. 管理者が企業を **コーチングマネジメント研修** プランで登録
2. クライアント C4 にその `companyId` を割り当て
3. C4 でホームを開く → **「セッション・研修ルーム」** が自動作成される
4. ルームを開くとパートナー欄は **「未決定」**
5. まだマッチ管理でパートナーを割り当てない

### T5 の作り方（コーチング・セッション種別 3 回構成の例）

企業設定で **総セッション数 = 3**、各回のフォーム種別:


| 回   | 種別              |
| --- | --------------- |
| 1回目 | ロールプレイ評価        |
| 2回目 | 通常のセッションフィードバック |
| 3回目 | ロールプレイ評価        |

### 5.9 1on1セッション — ロールプレイ評価（T5）【重点】

**前提:** コーチングプラン、当該回が「ロールプレイ評価」に設定されている

#### 入力前（相互開示前）


| #     | 手順            | 期待結果                                       |
| ----- | ------------- | ------------------------------------------ |
| RP-01 | クライアントのみ入力・保存 | パートナー側は **自分のフォームのみ** 見える（相手の点数・自由記述は見えない） |
| RP-02 | パートナーのみ入力・保存  | クライアント側も **自分のフォームのみ**                     |
| RP-03 | 管理者           | 双方の入力を閲覧できる（仕様どおり）                         |


#### 双方入力後


| #     | 手順                   | 期待結果                                               |
| ----- | -------------------- | -------------------------------------------------- |
| RP-04 | 双方が条件を満たして保存         | **「フィードバックが届きました」** 表示、双方の良かった点・改善点が開示             |
| RP-05 | レーダーチャート             | **「レーダーチャート（自己評価 ・パートナーからの評価）」** と表示（「ギャップ」表記ではない） |
| RP-06 | 各項目の点数               | チャートの下に双方の点数が表示される                                 |
| RP-07 | 通知                   | アプリ内通知 ＋ メールが双方に届く                                 |
| RP-08 | 開示後にクライアント/パートナーが再編集 | 不可（管理者は別途）                                         |


#### クライアント入力の完了条件

- 全項目の点数  
- セッション満足度（1〜10）  
- 満足度の理由

#### セッション種別の切り替え（T5 構成例）


| #     | 手順                 | 期待結果                           |
| ----- | ------------------ | ------------------------------ |
| RP-09 | 2回目（通常設定）のセッション画面  | ロールプレイではなく **通常の振り返りフォーム**     |
| RP-10 | 1回目・3回目            | ロールプレイ評価 UI                    |
| RP-11 | マッチルームの 1on1 一覧バッジ | ロールプレイ回は「自己評価」系、通常回は「振り返り」系の表示 |

## 6. ロール別・最短確認ルート（担当割り当て用）

デバッグ人数が少ないときは、以下を **1 人 1 ロール** で担当してください。


| 担当              | 最短ルート（30〜60分目安）                                                        |
| --------------- | ---------------------------------------------------------------------- |
| **クライアント担当**    | T4: 未割当ルーム → 準備タブ3つ → パートナー割当後 CP-08 → T5: ロールプレイ RP-01〜07 → 通常回 SE-01 |
| **パートナー担当**     | T1: CH-01〜02 → SC-01〜05 → SE-02 → T5: ロールプレイ入力 → 請求 IN-01              |
| **管理者担当**       | 企業・プラン設定 AD-01〜06 → マッチ AD-07〜09 → 請求 IN-03〜04 → T7 AD-10〜12           |
| **クライアント管理者担当** | T6: CA-01〜03                                                           |
| **回帰担当**        | セクション 4 スモーク ＋ 直近修正（下記セクション 7）を一覧チェック                                  |

## 7. 直近リリース分 — 重点回帰チェックリスト

以下は **必ず全員が一度は目を通す** 項目です（2025年6月時点の主要変更）。

- コーチングクライアントのホーム表記が **「セッション・研修ルーム」**
- コーチング・企業割当後、**パートナー未割当でもルームに入れる**
- 未割当中、概要・チャット・日程・1on1 はグレー、準備タブ3つは利用可
- 企業設定で **回ごとのフォーム種別**（ロールプレイ / 通常）が効く
- ロールプレイ **相互開示**（片方だけでは相手の入力が見えない）
- 開示後のレーダーチャート表記（自己評価 ・パートナーからの評価）
- 開示時のメール・通知
- 予約確定カレンダー文面（チャットメモ文言なし）
- マッチルームガイド「ここから始めてください」のみ
- 1on1フォーマットの文言・メモ欄の意味
- クライアントのグローバル FTA ナビが出ない（コーチングプラン）
- 45/35分セッションの候補グリッド・確定時刻調整

## 9. スコープ外・既知の注意


| 項目                     | 内容                                                       |
| ---------------------- | -------------------------------------------------------- |
| `mood-pdf-standalone/` | ローカル実験用。本番 QA 対象外                                        |
| Prisma のみの環境           | パートナー未割当ルーム作成は **Firebase 環境のみ** 対応。SQLite 単体では再現しない場合あり |
| 旧ロールプレイデータ             | `clientSubmittedAt` 導入前の保存データは、双方が再保存するまで開示されないことがある     |
| メール到達                  | 迷惑メール・ドメイン設定により遅延することがある                                 |
